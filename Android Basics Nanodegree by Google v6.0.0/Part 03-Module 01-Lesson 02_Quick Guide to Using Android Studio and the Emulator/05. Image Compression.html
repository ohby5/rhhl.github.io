05. Image Compression

Limited Device Memory

If you’ve ever bought a mobile device, you’ve likely encountered the question of how much memory to get. Devices, especially mobile devices, have a limited amount of memory which dictates how many applications run at a given time.

Image files can eat up a large chunk of the available memory. If your device runs out of memory, it can lead to nasty things happening such as your app crashing because there’s not enough for processes to run.

Drawable Resources that Use Bitmaps

How do we balance the need to load images on our devices with the limited memory available?

Let’s review and discuss how images are displayed on the screen.

We saw that we can set the source of an ImageView to be a Drawable Resource (Ex: @drawable/cake)

A Drawable resource can be generally described as any graphic that can be drawn on screen.

Every Drawable is stored as individual files in one of the res/drawable folders. Drawable resources come in a variety of different graphical file types but a common one is called Bitmap.

Bitmaps are graphic files that can have the extension .png, .jpg, or .gif and store images as a series of tiny squares called pixels.

Earlier, Katherine discussed with us the concept of Density Independent Pixels (Concept Video) which allow ImageViews to scale and resize appropriately across a wide range screen sizes and pixel densities.

Instead of using pixels, which vary in size and number across the different resolution devices, Android uses the unit dips so that the same bitmap of the image will appear the size no matter what device it’s displayed on.

Now if you’re wondering if this means we’ll need different sized images for each resolution then you’re correct!

In order for that one 48dp x 48dp image to appear the same size across devices, we’ll need to have a number of images at different pixel size.

Having several resolutions (the pixel count) of bitmaps available in your project means that when the device is running the app, only the bitmap specific to the device will be loaded. So instead of uploading the original image which could be huge and excessive, the only version of the image loaded is the lower pixel-count version of the same image re-sized in order to fit the memory limits and screen size of a mobile device.

This means in our res/ folder, the same image will have multiple dpi sizes. During run time when the image needs to be rendered, Android will determine which size is appropriate to pull up.
